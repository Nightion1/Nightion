================================================================================
  1. PROJECT OVERVIEW
================================================================================

Nightion is a fully offline AI coding assistant created by Nitin (solo 
developer). It runs entirely on the user's local machine with no cloud 
dependencies. The system uses the gemma4 language model served by Ollama to 
provide:

  - Conversational AI chat with reasoning transparency
  - Code generation and explanation
  - Data Structures & Algorithms (DSA) problem solving
  - Desktop application control (open/close apps natively on Windows)
  - Screen capture and vision-based code generation ("See & Code")
  - Human-like auto-typing of generated code into external editors
  - RAG (Retrieval-Augmented Generation) via ChromaDB vector store
  - Persistent memory with SQLite truth graph
  - Knowledge graph with confidence-based learning dynamics
  - Comprehensive telemetry and trace logging
  - Multi-mode interface: Chat Mode, Writer Mode, and Voice Mode

Key Design Principles:
  - Privacy-first: All processing happens locally. No data leaves the machine.
  - Offline-first: Every feature works without an internet connection.
  - Safety-first: Multi-layer security with guards, sandboxes, and capability
    gates that prevent destructive actions.
  - Streaming: Real-time token streaming over WebSocket for both reasoning and
    response tokens, giving users live visibility into the AI's thinking.

────────────────────────────────────────────────────────────────────────────────
  4.1  nightion_core.py — FastAPI Application & API Endpoints
────────────────────────────────────────────────────────────────────────────────

This is the main entry point of the application. It creates the FastAPI app 
instance, initializes all core singletons, and defines every HTTP/WebSocket 
endpoint.

Key Responsibilities:
  - Mounts the static file server at /static
  - Serves the main UI at GET /
  - WebSocket chat endpoint at /ws/chat (streaming AI responses)
  - Health check, ping, stats, config APIs
  - Screenshot capture and vision-code generation APIs
  - See & Code pipeline endpoint
  - Human-like typing endpoints (type-humanlike, type-anywhere)
  - Telemetry/trace retrieval APIs
  - Session history API

Singleton Objects (created at module load):
  - app:          FastAPI application instance
  - memory_db:    MemoryManager (JSON-based)
  - memory_core:  MemoryCore (SQLite-based)
  - router:       ToolRouter (semantic intent classification)
  - orchestrator: Orchestrator (query processing)

WebSocket Chat Flow (/ws/chat):
  1. Client sends JSON: { message, use_rag, session_id }
  2. Server generates a unique trace_id
  3. User message logged to MemoryCore
  4. Conversation history fetched for context
  5. AgentRequest constructed and passed to Orchestrator
  6. Streaming callback forwards think_token and token messages to client
  7. Final response logged and "done" signal sent

Run Command:
  python nightion_core.py
    OR
  uvicorn nightion_core:app --host 0.0.0.0 --port 8999 --reload


────────────────────────────────────────────────────────────────────────────────
  4.2  orchestrator.py — Query Orchestration & LLM Streaming
────────────────────────────────────────────────────────────────────────────────

The Orchestrator is the central intelligence coordinator. It receives queries 
from the WebSocket handler, builds conversation context, and streams responses 
from Ollama in real-time.

Class: Orchestrator
  __init__(router, max_retries=2, role=PRIMARY_ORCHESTRATOR)

Primary Method: execute_task(request, forced_intent, ui_feedback_cb)
  1. Extracts query and session_id from the request
  2. Fetches up to 8 recent messages from session history for context
  3. Logs the user message to MemoryCore
  4. Calls _call_ollama_streaming() with the assembled prompt
  5. Returns AgentResponse with the result

Streaming Pipeline: _call_ollama_streaming(query, history_context, ui_callback)
  - Constructs a full prompt with SYSTEM_PROMPT + conversation history + query
  - Sends to Ollama at http://localhost:11434/api/generate with:
      stream=True, think=True (native think-mode)
  - Reads NDJSON lines from the streaming response
  - For each line:
      "thinking" field → forwarded as {type: "think_token", content: ...}
      "response" field → forwarded as {type: "token", content: ...}
      Inline <think>...</think> tags are extracted and forwarded separately
  - Assembles the final result combining thinking and response buffers

System Prompt:
  "You are Nightion, a powerful AI coding assistant created by Nitin 
   (solo developer). Your underlying model is gemma4 running entirely 
   offline via Ollama..."

Error Handling:
  - ConnectionRefusedError → "Ollama is not running" message
  - Empty response → "Model returned an empty response" warning
  - General exceptions → descriptive error with troubleshooting steps


────────────────────────────────────────────────────────────────────────────────
  4.3  tool_router.py — Semantic Intent Classification
────────────────────────────────────────────────────────────────────────────────

Replaces traditional regex/if-else routing with cosine-similarity-based 
semantic classification using the all-MiniLM-L6-v2 sentence-transformer model.

Classification Pipeline (in priority order):
  1. Hard Rule — Greeting:     Exact word match (hi, hello, hey, etc.)
  2. Hard Rule — Browser:      DOM/navigation keywords (navigate to, click button)
  3. Hard Rule — App Control:  OS verb + known app name (open notepad)
  4. Semantic Similarity:      Vector cosine distance against seeded examples
  5. Final Fallback:           Defaults to GENERAL intent

Supported Intents (IntentEnum):
  - GREETING            — Simple greetings
  - CODE                — Code generation requests
  - DSA                 — Data structure/algorithm explanations
  - GENERAL             — Conversational/knowledge questions
  - APP_CONTROL         — Desktop application management
  - BROWSER_AUTOMATION  — Browser interaction commands

Safety Features:
  - APP_CONTROL false-positive blocker: If vector similarity classifies a query
    as APP_CONTROL but it lacks both an OS verb AND a known app name, it's
    downgraded to GENERAL. Prevents "write the formula for gravity" from 
    opening an app.
  - Anaphoric/possessive skip: Patterns like "gravity's formula" or "Newton's 
    law" skip APP_CONTROL entirely to avoid false triggers.
  - Destructive action detection: is_destructive_action() checks for keywords 
    like delete, erase, format, purchase, etc.

Vector Store Seeding:
  - On first use, ~100+ canonical intent examples are embedded and stored in 
    the ChromaDB "intents" collection
  - Examples are re-seeded if the count changes (new examples added)

RouterConfig:
  - confidence_threshold: 0.35 (minimum cosine similarity to trust)
  - use_llm_fallback: False (replaced by vector similarity)


────────────────────────────────────────────────────────────────────────────────
  4.4  llm_adapter.py — Hybrid RAG Intelligence Brain
────────────────────────────────────────────────────────────────────────────────

The LLM adapter provides a two-layer hybrid intelligence system:

Layer 1: Local Ollama + RAG
  - Retrieves relevant chunks from ChromaDB vector store
  - Injects chunks as CONTEXT into the Ollama system prompt
  - Topic guard: only includes chunks whose topic keywords overlap with query
  - Strips identity echo lines the model sometimes parrots from the system prompt

Layer 2: Smart Fallback (Ollama offline)
  - ChromaDB vector search with min_score=0.60
  - Deterministic rules for: identity, frustration, math, time/date
  - If no offline answer → returns None → triggers web search escalation

Key Features:
  - Runtime State Injection: Every LLM call samples live system state
    (Ollama status, connectivity, model name) and injects it as a
    [RUNTIME STATE] block into the system prompt
  - Mode Question Interception: Questions like "are you online?" are answered
    deterministically from runtime state BEFORE reaching the LLM
  - Two-tier mode question detection:
      Tier 1: Unambiguous multi-word phrases ("are you online or offline")
      Tier 2: Keyword + question signal co-occurrence ("which model handles this")

Class: LocalizedLLMAdapter
  - generate_structured_thought(): Main entry point for all intents
  - _try_ollama_conversational(): GENERAL intent — conversational with RAG
  - _try_ollama(): CODE/DSA/SEARCH — JSON planner with RAG
  - _smart_fallback(): Fallback when Ollama is offline
  - _retrieve_context(): RAG retrieval with topic guard
  - _strip_identity_echo(): Removes echoed identity text from responses

  - get_all_language_preferences(): Full preference map across all categories
  - add_episodic_trace():         Record task outcomes for learning
  - add_verified_fact():          Store verified facts with confidence
  - fetch_recent_patterns():      Retrieve learned strategies

Database Path: memory/nightion_memory.db

────────────────────────────────────────────────────────────────────────────────
  5.3  knowledge_base.py — Topic-Hashed Knowledge Store
────────────────────────────────────────────────────────────────────────────────

SQLite-based knowledge store using SHA-256 topic hashing for O(1) lookups.
Stores code snippets, explanations, and source attribution.

Table Schema (knowledge):
  topic_hash, topic, language, code, explanation, source_url, source_name,

Trusted Sources:

Methods:
  - lookup(topic, language):       O(1) hash-based retrieval
  - store(topic, code, ...):       Persist learned knowledge
  - fuzzy_search(query, limit):    Keyword-based partial matching
  - is_trusted_source(url):        Validate source trustworthiness

Database Path: knowledge.db


────────────────────────────────────────────────────────────────────────────────
  5.4  knowledge_graph.py — Concept Graph with Confidence Dynamics
────────────────────────────────────────────────────────────────────────────────

A navigable concept graph stored in SQLite with bidirectional edges between 
related knowledge nodes.

Tables (added to nightion_memory.db):
  knowledge_nodes: id, concept, summary, example, related_topics_json,
                   learned_at, last_used, use_count, confidence, source_url
  knowledge_edges: from_id, to_id, relationship_type, created_at

Confidence Dynamics:
  - On save: confidence = value from Processor (0.2–1.0)
  - On successful use: confidence += 0.1
  - On failed/corrected use: confidence -= 0.2
  - Range clamped to [0.0, 2.0]

Deduplication (Three-Path Logic):
  Path A: Concept exists with high confidence (>=0.5) → bump last_seen only
  Path B: Concept exists with low confidence (<0.5) → overwrite with fresh data
  Path C: New concept → save as new node

Key Functions:
  - save_node():               UPSERT a knowledge node
  - save_node_deduped():       Smart deduplication before save
  - link_related():            Create bidirectional edges for related topics
  - search_nodes():            Keyword-based search with relevance scoring
  - is_unknown_topic():        Check if a topic is confidently known
  - update_node_confidence():  Adjust confidence with clamping
  - get_connected_nodes():     Graph traversal for related concepts
  - get_learned_patterns_count(): Live DB counter for UI display
  - get_total_chunks():        Total knowledge nodes count


────────────────────────────────────────────────────────────────────────────────
  5.5  vector_store.py — ChromaDB Vector Intelligence
────────────────────────────────────────────────────────────────────────────────

Real vector storage and retrieval using ChromaDB with sentence-transformers 
(all-MiniLM-L6-v2) for semantic similarity.

Collections:
  - "knowledge":  Scraped/learned content (code, explanations, articles)
  - "intents":    Canonical routing examples per IntentEnum value

Core Operations:
  - add(id, text, metadata, collection):    Embed and store text
  - search(query, top_k, min_score):        Cosine similarity search
  - count(collection):                      Real vector count
  - delete(id, collection):                 Remove a vector

Intent Routing:
  - seed_intents(intent_examples):  Embed canonical examples for classification
  - classify_intent(query):         Nearest-neighbour classification

Text Chunking:
  - chunk_text(text, max_tokens):   Split text into overlapping ~300-word chunks

Similarity Scoring:
  ChromaDB cosine distance → similarity: sim = 1 - dist/2

Singleton Access: get_vector_store() returns the global VectorStore instance
Persistent Storage: chroma_db/ directory


────────────────────────────────────────────────────────────────────────────────
  5.6  context_injector.py — Knowledge Context Injection
────────────────────────────────────────────────────────────────────────────────

On every query, searches the knowledge graph for relevant concepts, ranks them, 
and injects the top 3 entries into the LLM prompt as learned context.

Ranking Formula:
  score = keyword_matches × confidence × log(1 + use_count)

Confidence Update Signals:
  "success" → +0.1  (knowledge was referenced in response)
  "ignored" → -0.2  (knowledge was relevant but contradicted/ignored)
  "failed"  → -0.1  (code generation failed entirely)

Signal Inference:
  - Empty/error response → "failed"

Isolated execution environment for running Python code safely.

Security Measures:

  3. Execute via subprocess with capture_output=True
  4. Parse return code, stdout, stderr

Four capability levels controlling what tools can execute:

  6.6  verifier.py — Tri-State Verification Layer
────────────────────────────────────────────────────────────────────────────────

The absolute authority validation layer that grades logical correctness and 
prevents hallucinations. Returns explicit tri-state decisions.

Verification Pipeline:
  1. Semantic Intent Validation — catches intent mismatches
     (e.g., searching when code generation was requested)
  2. Immediate Failure Checks — FAILED or BLOCKED status
  3. Malformed/Empty Validation — null/empty output detection
  4. Chaos/Uncertainty Checks:
     a. Stub Detection: catches placeholder/mock responses
        ("execution complete", "simulated verification", "not implemented")
     b. Confusion Detection: uncertain language in output
  5. Success Endorsement (confidence: 0.90)

Decisions: PASS, FAIL, UNCERTAIN
Severities: low, medium, high

Semantic Mismatch Detection:
  - Catches "Search-instead-of-Code" anti-pattern: when a coding query
    (with verbs like "write", "implement" + languages like "python", "java")
    gets routed to search instead of code generation

  8.2  see_and_code.py — Vision-Based Code Generation & Auto-Typing
────────────────────────────────────────────────────────────────────────────────

The "See & Code" pipeline: screenshot → AI vision analysis → code extraction 
→ human-like typing into external editors.

Four Components:

1. Screen Capture (capture_screen):
   - Alt+Tab to target window first
   - Screenshot via pyautogui
   - Alt+Tab back to Nightion
   - Resize to max 1280px width
   - Returns base64 PNG string

2. Multimodal Code Generation (generate_code_from_image):
   - Sends screenshot to gemma4 vision model via Ollama
   - Prompt instructs: "Read the problem from the screenshot, write COMPLETE 
     WORKING solution, ONLY code in a fenced block"
   - Two attempts: concise prompt first, then detailed retry with higher 
     num_predict
   - Strips <think>...</think> tags from model output
   - Extracts code from fenced code blocks (language-specific → generic → raw)
   - Supports 10 languages: Python, C++, C, Java, JavaScript, TypeScript, C#,
     Go, Rust, SQL

3. Human-Like Typing (type_code_humanlike):
   - Types char-by-char using pynput keyboard controller
   - Realistic timing: 35-70ms base, extra delay for newlines/brackets
   - Burst mode: ~8% chance to type 5-8 chars at 10-20ms (fast typing)
   - Typo simulation: ~4% chance after 50+ chars, QWERTY neighbor keys,
     followed by backspace and correction
   - Protected characters (;{}[]()=<>:, etc.) never get typo'd
   - Protected keywords (def, class, return, etc.) never get typo'd
   - Indentation stripping for online editor compatibility
   - Cancel support via threading.Event

4. External Click Listener (wait_for_external_click_then_type):
   - Uses pynput mouse listener to wait for user click
   - After click detected, begins human-like typing
   - Configurable timeout (default: 60 seconds)
   - Supports cancellation

================================================================================
  17. DATA FLOW WALKTHROUGH
================================================================================

Complete flow for a user chat message:

  1. User types message in browser UI (static/app.js)
  2. app.js sends JSON via WebSocket to /ws/chat
  3. nightion_core.py receives the message
     a. Generates unique trace_id (UUID4)
     b. Logs user message to MemoryCore.session_chat
     c. Fetches last 10 messages of session history
     d. Constructs AgentRequest
  4. Orchestrator.execute_task() is called
     a. Fetches 8 recent messages for conversation context
     b. Logs user message again to its own MemoryCore
     c. Builds full prompt: SYSTEM_PROMPT + history + query
  5. Orchestrator._call_ollama_streaming() sends to Ollama
     a. HTTP POST to http://localhost:11434/api/generate
     b. stream=True, think=True, temperature=0.7, num_predict=2048
  6. Ollama streams NDJSON lines back
     a. Each line parsed for "thinking" and "response" fields
     b. thinking tokens → ui_callback({type: "think_token", ...})
     c. response tokens → ui_callback({type: "token", ...})
     d. Inline <think>...</think> tags extracted and forwarded
  7. ui_callback (in nightion_core.py) forwards to WebSocket
     a. websocket.send_json({type: "think_token", content: ...})
     b. websocket.send_json({type: "token", content: ...})
  8. app.js receives tokens in real-time
     a. think_tokens displayed in collapsible reasoning block
     b. tokens accumulated and rendered as markdown
  9. When Ollama sends done=true:
     a. Orchestrator assembles final result string
     b. If thinking was present: "<think>...</think>\n\nresponse"
     c. Response logged to MemoryCore.session_chat
  10. nightion_core.py sends final "done" message to WebSocket
  11. app.js handles "done": finalizes rendering, enables input
